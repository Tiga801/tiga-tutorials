# format-markdown

对一个或多个本地 Markdown 文件执行标准格式规范化。

- **单文件**：读取 → 检查 → 原地重写为标准格式
- **多文件**：读取全部 → 理解内容 → 合成一份新文档（原文件不改动）

## 执行流程

### 单文件

1. `Read` 读取文件
2. 对照格式规则清单逐项检查
3. 应用文档骨架重写（保留全部技术内容，仅改格式）
4. `Write` 原地覆写
5. 输出：列举修正的问题（简短，3-5 条）

### 多文件

1. `Read` 读取全部文件
2. 识别共同主题和内容重叠度
3. `AskUserQuestion` 确认：
   - 输出文件名（提供建议默认值）
   - 保存目录（`tutorials/` 或 `blog-summaries/`）
4. 按合并策略生成新文档（见下方）
5. `Write` 写入新路径，**不修改原始文件**

### 多文件合并策略

| 情形 | 处理方式 |
|------|----------|
| 相同章节（如"Docker 权限配置"） | 合并为一节，用表格或 blockquote 标注平台差异 |
| 互补章节（内容不重叠） | 顺序合并，保持逻辑流 |
| 冲突内容（同一操作做法不同） | 两者都保留，`> **来源：** 文件名` 注明出处 |

## 标准文档骨架

````markdown
# <标题>

> **环境**：<系统 / 语言 / 版本>（适用时填写）
> **最后更新**：<YYYY-MM-DD>

## 概述

<2-4 句：解决什么问题，适用场景。不使用对话式开场白。>

## 前提条件（可选）

- 依赖项或已知前提

## <步骤 / 主要章节>

### <子章节>

> **注意：** 重要提醒使用 blockquote + 粗体标题

```bash
# 每个代码块必须标注语言
command here
```

## 注意事项（可选，汇总所有 warning / gotcha）

## 参考资料（可选）

- [描述](URL)
````

## 格式规则清单

| 规则 | 要求 |
|------|------|
| H1 标题 | 全文唯一，第一行，来自文章主题 |
| 标题层级 | 不跳级：`#` → `##` → `###` → `####` |
| 代码块语言 | 每个 ` ``` ` 必须标注语言（`bash` `yaml` `python` `json` 等）|
| 内联代码 | 命令、文件名、配置键名、变量名用 `` ` `` 包裹 |
| 粗体 | 关键术语首次出现；警告/注意标题（`**注意：**`）|
| 元数据区块 | H1 后接 blockquote，列出环境、日期等元信息 |
| 对话式语气 | 删除（"这是一份为你整理的..."、"需要我帮你..."、"下一步建议："类内容）|
| 水平分割线 | 仅用于明确的主题分隔，不在每节末尾滥用 `---` |
| 空行 | 标题前后各留一空行；代码块前后各留一空行 |

## 内容类型判断

| 特征 | 文档类型 | 适用骨架节顺序 |
|------|----------|----------------|
| 步骤序号、命令、配置、代码为主 | 教程 | 概述 → 前提条件 → 步骤 → 注意事项 → 参考资料 |
| 概念分析、观点、经验总结为主 | 博客总结 | 核心宗旨 → 关键概念 → 总结 → 参考资料 |

## 常见错误

- **删除技术内容**：只改格式，不删减任何命令、配置、代码或注意事项
- **H1 位置错误**：H1 必须是文件第一行内容，元数据 blockquote 紧随其后
- **合并时修改原文件**：多文件模式下原始文件必须保持不变
- **代码块无语言标注**：哪怕是单行 shell 命令也要写 ` ```bash `
